## 問題情境

很多小組在第一週會遇到一個問題：每一份文件單獨看好像都可以，但放在一起就不一致。

常見情況包括：

1. 問題陳述說要服務「學生與行政人員」，但利害關係人表只列學生。
2. 範圍內項目寫了「完整金流」，但可行性分析說 5 週做不完。
3. 原型核心任務只有「登入」與「查詢」，但題目其實是流程改善系統。
4. 風險清單說「時間不夠」，但沒有把任何功能移到範圍外。
5. 期末想展示的原型任務，無法追溯回問題陳述與利害關係人需求。

今天的重點不是增加更多內容，而是讓內容互相對得起來。

## 核心概念

### 概念 1：專案章程是整個專案的合約

專案章程（Project Charter）不是作文，也不是題目介紹。它是小組對這個專案的共同承諾，至少要回答：

1. 我們要解決誰的問題？
2. 目前情境是什麼？
3. 痛點與影響是什麼？
4. 本期要做到哪些範圍？
5. 本期不做哪些範圍？
6. 期末要展示哪三個核心任務？
7. 成功的判斷標準是什麼？
8. 主要風險是什麼？

如果專案章程寫不清楚，後面的需求文件包（Requirements Package）、軟體需求規格書（SRS）、軟體設計文件（SDD）和原型（prototype）都會不穩。

### 概念 2：專案章程第 2 版（Project Charter v2）要比第 1 版更具體

專案章程第 1 版（Project Charter v1）通常是初步想法；專案章程第 2 版（Project Charter v2）必須加入 7/7 的檢核結果。

| 項目       | v1 常見狀態      | v2 應達到狀態                                    |
| ---------- | ---------------- | ------------------------------------------------ |
| 專案題目   | 大致知道方向     | 清楚指出目標使用者與核心流程                     |
| 問題陳述   | 有痛點但可能籠統 | 有目標使用者、目前情境、痛點、改善方向與範圍焦點 |
| 利害關係人 | 只列主要使用者   | 同時列出使用者、審核者、管理者、受影響者         |
| 範圍內項目 | 想做的功能很多   | 只保留 5 週內能分析、設計與展示的項目            |
| 範圍外項目 | 寫得太少         | 明確排除正式部署、金流、跨系統整合等高風險項目   |
| 原型任務   | 只有功能名稱     | 有任務起點、任務終點與初步驗收標準               |
| 風險       | 太籠統           | 每項風險都有應對方式                             |

### 概念 3：短講提案是用來檢查專案清楚度

短講提案（project pitch）不是期末報告，而是用 3 分鐘確認專案是否能被別人理解。

一個清楚的短講提案需要回答：

1. 我們的系統名稱是什麼？
2. 誰遇到問題？
3. 現在怎麼做？痛點是什麼？
4. 本系統要改善什麼？
5. 本期範圍內做什麼？
6. 本期範圍外不做什麼？
7. 期末原型會展示哪三個任務？
8. 目前最大的風險是什麼？

如果 3 分鐘說不清楚，通常代表專案章程還不夠清楚。

### 概念 4：同儕提問可以用來找盲點

同儕提問（peer questions）不是批評，而是幫小組找出沒有想清楚的地方。

好的提問通常會問：

1. 這個使用者真的會使用嗎？
2. 誰有權限決定這件事？
3. 這個功能 5 週內是否太大？
4. 這個流程的例外情況是什麼？
5. 這項需求是否能驗收？
6. 如果不做某個功能，系統是否仍然有價值？

### 概念 5：期末原型任務要從第一週就定出來

原型展示任務（prototype demo tasks）是期末要展示的核心流程，不是隨便做幾個畫面。

好的原型任務需要包含：

1. 主要角色。
2. 任務起點。
3. 操作流程。
4. 任務終點。
5. 驗收標準。

範例：

| 任務                   | 主要角色     | 任務起點               | 任務終點                 | 初步驗收標準                         |
| ---------------------- | ------------ | ---------------------- | ------------------------ | ------------------------------------ |
| 建立活動並送出場地申請 | 社團幹部     | 社團幹部登入活動管理頁 | 活動與場地申請送出待審核 | 系統顯示待審核狀態，並可在清單中查看 |
| 報名活動並進入候補     | 一般學生     | 學生看到活動列表       | 報名額滿後進入候補名單   | 系統顯示候補順位與通知狀態           |
| 審核場地申請           | 課外組承辦人 | 承辦人查看待審核清單   | 核准或退回場地申請       | 系統更新申請狀態並留下審核紀錄       |

## 範例解析

### 範例 1：從 v1 修到 v2

v1 題目：

```
社團系統
```

問題：

1. 不知道是給誰用。
2. 不知道處理哪個流程。
3. 不知道期末展示什麼。

v2 題目：

```
面向學生社團幹部、一般學生與課外組承辦人的活動報名、候補與場地申請流程整合系統
```

改善：

1. 指出目標角色。
2. 指出核心流程。
3. 限定範圍，避免變成完整校務系統。

### 範例 2：範圍內與範圍外的修正

原本範圍內：

1. 活動報名。
2. 場地申請。
3. 線上付款。
4. 學校單一登入。
5. 手機 App。

修正後：

| 類型                   | 項目                                                                              |
| ---------------------- | --------------------------------------------------------------------------------- |
| 範圍內（in-scope）     | 活動建立、活動報名、候補、場地申請、審核狀態                                      |
| 範圍外（out-of-scope） | 正式金流、學校單一登入、手機原生應用程式（native mobile app）、正式部署、跨校活動 |

修正理由：

```
線上付款、學校單一登入與手機原生應用程式會讓技術與時程風險過高。期末原型只需要展示核心流程，不需要正式部署。
```

### 範例 3：三分鐘短講提案稿

```
我們小組的題目是「面向學生社團幹部、一般學生與課外組承辦人的活動報名、候補與場地申請流程整合系統」。

目前社團活動資訊常分散在 Google 表單、訊息群組與人工紀錄中，造成活動報名狀態不清楚、候補通知延遲、場地申請與活動資訊不同步。

本系統希望協助社團幹部建立活動與申請場地，讓一般學生完成報名或候補，並讓課外組承辦人審核場地申請。

本期範圍內包含活動建立、報名、候補、場地申請與審核狀態。範圍外包含正式金流、學校單一登入、正式部署與手機原生應用程式。

期末原型會展示三個任務：第一，社團幹部建立活動並送出場地申請；第二，一般學生報名活動並在額滿時進入候補；第三，課外組承辦人審核場地申請。

目前最大風險是場地審核規則可能過於複雜，因此本期先以簡化規則建立原型，正式行政簽核流程放入範圍外。
```

### 範例 4：同儕提問與修正

| 同儕問題                            | 小組回應                                   | 是否修正專案章程 |
| ----------------------------------- | ------------------------------------------ | ---------------- |
| 候補通知是用 email 還是系統內通知？ | 本期原型只展示系統內通知狀態，不串接 email | 是，補入範圍外   |
| 場地衝突怎麼判斷？                  | 以日期、時段、場地三個欄位判斷衝突         | 是，補入核心規則 |
| 活動是否需要老師審核？              | 本期不處理老師審核，只列為後續擴充         | 是，補入範圍外   |
