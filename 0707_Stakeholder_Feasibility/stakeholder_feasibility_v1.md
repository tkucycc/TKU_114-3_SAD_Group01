## 問題情境

假設你的題目是「社團活動報名與場地申請系統」。7/6 可能已經寫出這樣的問題陳述：

```
對於學生社團幹部、一般學生與課外組承辦人而言，目前活動建立、學生報名、候補通知與場地申請分散在不同表單、訊息群組與人工紀錄中，常造成資訊不同步、場地衝突與候補通知延遲。
本專案希望設計一個社團活動報名與場地申請流程整合系統，協助社團幹部建立活動、學生完成報名與候補、課外組進行場地審核。本期聚焦於活動建立、報名候補、場地申請與審核流程，不處理正式金流與學校單一登入串接。
```

這段問題陳述看起來已經比「我要做社團系統」清楚，但仍然不夠完整。因為你還需要知道：

1. 誰會每天使用這個系統？
2. 誰可以批准或否決系統流程？
3. 誰提供資料？
4. 誰會因為系統改變而增加工作？
5. 誰可能反對這個系統？
6. 5 週內哪些部分真的做得出原型（prototype）？
7. 哪些功能應該放到範圍外（out-of-scope）？

這些問題就是今天的重點。

## 核心概念

### 概念 1：使用者不等於利害關係人

使用者（users）是會直接操作系統的人。利害關係人（stakeholders）是會受到系統影響、能提供需求、能阻礙專案、能批准流程，或需要從系統取得資訊的人。

以社團活動報名與場地申請系統為例：

| 角色         | 是否為使用者（user） | 是否為利害關係人（stakeholder） | 原因                                         |
| ------------ | -------------------- | ------------------------------- | -------------------------------------------- |
| 一般學生     | 是                   | 是                              | 會報名活動、取消報名、查看候補狀態           |
| 社團幹部     | 是                   | 是                              | 會建立活動、管理報名、申請場地               |
| 課外組承辦人 | 是                   | 是                              | 會審核活動與場地申請                         |
| 場地管理員   | 可能是               | 是                              | 需要知道場地使用狀態與衝突                   |
| 指導老師     | 可能不是             | 是                              | 可能需要確認活動資訊或負責督導               |
| 學校資訊處   | 可能不是             | 是                              | 若涉及帳號、權限或系統整合，會影響技術可行性 |

如果只看「誰會登入系統」，就會漏掉很多重要需求。

### 概念 2：利害關係人地圖要呈現關係，不只是列名單

利害關係人地圖（stakeholder map）不是把角色全部列出來而已，而是要說明：

1. 誰是主要使用者？
2. 誰提供資料？
3. 誰審核？
4. 誰接收通知？
5. 誰管理規則？
6. 誰可能反對或限制系統設計？

可以用文字、表格或圖表示。今天至少要完成表格版；如果工具允許，也可以補一張圖。

### 概念 3：權力 / 關注矩陣

權力 / 關注矩陣（power-interest matrix）用來判斷每個利害關係人要怎麼處理：

| 類型           | 特徵                           | 對應策略                 |
| -------------- | ------------------------------ | ------------------------ |
| 高權力、高關注 | 能影響專案，也很在意結果       | 密切管理，需求要優先確認 |
| 高權力、低關注 | 能影響專案，但平常不一定參與   | 定期告知，避免最後才反對 |
| 低權力、高關注 | 很常使用或受影響，但決策權較低 | 深入訪談，蒐集真實痛點   |
| 低權力、低關注 | 影響較小                       | 保持基本通知即可         |

範例：

| 角色         | 權力 | 關注 | 分類           |
| ------------ | ---- | ---- | -------------- |
| 課外組承辦人 | 高   | 高   | 高權力、高關注 |
| 一般學生     | 低   | 高   | 低權力、高關注 |
| 學校資訊處   | 高   | 低   | 高權力、低關注 |
| 路過活動訪客 | 低   | 低   | 低權力、低關注 |

### 概念 4：可行性分析不是說「我們做得到」而已

可行性分析（feasibility analysis）要判斷專案在限制條件下是否合理。今天至少要分析四種可行性：

| 類型                                  | 中文說明                 | 要問的問題                                               |
| ------------------------------------- | ------------------------ | -------------------------------------------------------- |
| 技術可行性（technical feasibility）   | 工具與技術是否支援       | 我們用現有工具能不能做出可展示原型？需要串接正式系統嗎？ |
| 操作可行性（operational feasibility） | 使用情境與流程是否可運作 | 使用者會不會願意用？流程是否比原本更清楚？               |
| 時程可行性（schedule feasibility）    | 5 週內是否能完成         | 需要做的內容是否太多？本期能展示哪三個核心任務？         |
| 經濟可行性（economic feasibility）    | 成本與效益是否合理       | 專案是否需要付費工具、伺服器或大量人力？                 |

在本課中，「不可行」不是失敗，而是提醒你要縮小範圍。好的小組會把不可行部分放入範圍外（out-of-scope）。

### 概念 5：風險清單要連到應對方式

風險（risk）不是抱怨，而是可能讓專案失敗的事件。每個風險都要寫出影響、可能性與應對方式。

建議格式：

| 風險                                                       | 影響                     | 可能性 | 應對方式                               |
| ---------------------------------------------------------- | ------------------------ | ------ | -------------------------------------- |
| 題目範圍太大                                               | 期末無法完成原型         | 高     | 將金流、正式登入、跨系統串接放入範圍外 |
| 找不到真實使用者訪談                                       | 需求可能不準確           | 中     | 使用模擬訪談與同儕角色扮演補足初步需求 |
| 組員工具能力不足                                           | 模型與原型品質不穩       | 中     | 指定一種工具，先完成低擬真原型         |
| 生成式 AI 產出過度通用（generative AI output too generic） | 文件像模板，無法對應題目 | 中     | 要求每項建議都回到本組情境檢查         |

## 範例解析

### 範例 1：從題目找出利害關係人

題目：

```
校園二手書交易系統
```

可能利害關係人：

| 角色       | 與系統的關係             | 可能需求                         |
| ---------- | ------------------------ | -------------------------------- |
| 賣家       | 刊登書籍、管理交易狀態   | 快速刊登、修改售價、標記已售出   |
| 買家       | 搜尋書籍、收藏、聯絡賣家 | 找到指定課本、比較價格、預約面交 |
| 系統管理者 | 處理違規與爭議           | 查看檢舉、停權不當帳號           |
| 課程學生   | 可能需要特定課本         | 依課程名稱或書名搜尋             |
| 學校單位   | 可能關心交易安全         | 避免詐騙、保護個資               |

這裡不能只寫「使用者」，因為賣家和買家的任務不同；也不能漏掉管理者，因為二手交易可能有檢舉與爭議問題。

### 範例 2：權力 / 關注矩陣

以校園二手書交易系統為例：

| 角色       | 權力 | 關注 | 分類           | 對應策略                   |
| ---------- | ---- | ---- | -------------- | -------------------------- |
| 買家       | 低   | 高   | 低權力、高關注 | 深入了解搜尋與交易痛點     |
| 賣家       | 低   | 高   | 低權力、高關注 | 確認刊登流程是否簡單       |
| 系統管理者 | 高   | 中   | 高權力、中關注 | 確認違規處理與權限需求     |
| 學校單位   | 高   | 低   | 高權力、低關注 | 說明不做正式金流與校務整合 |

### 範例 3：可行性分析

題目：

```
校園二手書交易系統
```

| 類型                                  | 分析                                                                                                 |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| 技術可行性（technical feasibility）   | 可用 Figma 介面設計工具（Figma）或網頁原型工具做出刊登、搜尋、收藏、預約面交流程，不需要正式付款系統 |
| 操作可行性（operational feasibility） | 學生有二手書交易需求，但需要降低詐騙與資訊不完整問題                                                 |
| 時程可行性（schedule feasibility）    | 5 週內可展示刊登、搜尋、預約與檢舉流程；不適合加入完整金流                                           |
| 經濟可行性（economic feasibility）    | 可使用免費工具完成原型，不需要伺服器或付費資料庫                                                     |

結論：

```
本期可行，但需將金流、物流、正式身分驗證與跨校交易列為範圍外（out-of-scope）。
```

### 範例 4：風險清單

| 風險                     | 影響         | 可能性 | 應對方式                                   |
| ------------------------ | ------------ | ------ | ------------------------------------------ |
| 加入付款功能導致範圍過大 | 期末無法完成 | 高     | 本期只做面交預約，不做金流                 |
| 書籍資料格式不一致       | 搜尋體驗差   | 中     | 規定刊登欄位：書名、作者、課程、價格、狀態 |
| 缺少真實交易資料         | 需求不夠具體 | 中     | 用同儕訪談與模擬資料建立初步情境           |
| 原型連結權限未開         | 期末無法展示 | 低     | 繳交前用不同帳號測試連結                   |

## 常見錯誤與診斷

| 常見錯誤                                                 | 診斷問題                                       | 修正方式                                       |
| -------------------------------------------------------- | ---------------------------------------------- | ---------------------------------------------- |
| 只列使用者，沒有列利害關係人                             | 是否漏掉審核者、管理者、資料提供者、受影響者？ | 補上直接與間接受影響的角色                     |
| 每個角色都寫高權力、高關注                               | 是否真的每個人都能影響專案決策？               | 重新判斷權力與關注                             |
| 可行性分析全部寫可行                                     | 是否有具體工具、時程與限制判斷？               | 每一類至少寫出一個風險                         |
| 風險只寫「時間不夠」                                     | 是否說明造成時間不夠的原因？                   | 改成具體風險，例如範圍過大、工具不熟、需求不清 |
| 沒有調整範圍                                             | 做完可行性分析後，範圍內 / 範圍外沒有任何變化  | 至少檢查一項是否應移到範圍外                   |

## 形成性評量

### 檢核 1：概念題

請回答：

```text
為什麼使用者（users）不一定等於利害關係人（stakeholders）？
```

可接受答案要包含：使用者會直接操作系統；利害關係人可能不操作系統，但會被系統影響、提供需求、審核流程或影響專案決策。

### 檢核 2：判斷題

如果某角色不會登入系統，就不需要放進利害關係人分析。

```text
對 / 錯
```

標準答案：錯。理由是有些角色不登入系統，但仍會提供規則、資料、審核或限制。

### 檢核 3：小組口頭檢核

每組用 1 分鐘說明：

1. 本組最高權力、高關注的利害關係人是誰？
2. 本組最大的可行性風險是什麼？
3. 今天新增了哪一個範圍外項目？

### 檢核 4：離場回饋

下課前每組需完成：

1. 利害關係人分析表。
2. 權力 / 關注矩陣。
3. 可行性分析。
4. 風險清單第 1 版。
