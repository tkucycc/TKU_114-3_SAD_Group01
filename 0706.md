本單元是「系統分析與設計」第一天完整教材，對應 115/07/06（一）02-04 與 06-08。今天的任務是理解這門課不是只畫統一塑模語言（UML）圖，也不是只做期末簡報，而是以一個小組專案貫穿 5 週，逐步完成：

7. 測試案例（Test Cases）
8. 期末專案交付包（Final Package）

今天結束前，每組要選定一個可在暑修期間完成分析、設計與原型展示（prototype demo）的題目，並完成專案章程第 1 版（Project Charter v1）的核心欄位。

## 問題情境

假設學校想做一個「學生社團活動報名與場地申請系統」。一開始需求看起來很簡單：

```
學生可以報名活動，社團可以申請場地，學校可以審核。
```

但實際討論後會發現很多問題：

1. 誰可以建立活動？社團幹部、指導老師、課外組，還是所有學生？
2. 場地衝突怎麼處理？先申請先贏，還是依活動性質排序？
3. 報名人數額滿後能不能候補？
4. 學生取消報名後，候補者如何通知？
5. 活動涉及保險、收費或校外人士時，流程是否不同？
6. 誰可以看到學生個資？
7. 系統要不要整合學校帳號、電子郵件（email）、iClass 教學平台（iClass）或行事曆？
8. 如果活動取消，已報名學生、場地管理員與社團幹部如何同步知道？

這些問題不是單純使用者介面（UI）或程式語法問題，而是系統分析與設計要處理的問題。

## 核心概念

### 概念 1：系統分析與設計不是先畫圖，而是先釐清問題

系統分析與設計的第一步不是打開工具畫圖，而是回答：

1. 現在的問題是什麼？
2. 誰受到這個問題影響？
3. 目前流程怎麼運作？
4. 哪些資料被建立、讀取、修改、刪除？
5. 系統要支援哪些決策？
6. 哪些事情明確不在本系統範圍？
7. 怎麼判斷系統設計是成功的？

如果一開始問題定義錯誤，後面的使用案例（use case）、實體關係圖（ERD）、類別圖（class diagram）、原型（prototype）會一起偏掉。

### 概念 2：資訊系統包含人、流程、資料、規則與技術

| 面向               | 引導問題                                   |
| ------------------ | ------------------------------------------ |
| 人員（People）     | 誰使用？誰管理？誰審核？誰被影響？         |
| 流程（Process）    | 現在流程是什麼？系統後流程會變成什麼？     |
| 資料（Data）       | 需要哪些資料？資料從哪裡來？誰能修改？     |
| 規則（Rules）      | 有哪些限制、權限、例外、審核規則？         |
| 技術（Technology） | 要用哪些工具展示？是否需要串接既有系統？   |
| 品質（Quality）    | 系統要安全、易用、可靠、可維護到什麼程度？ |

### 概念 3：全期專案交付物是一條可追蹤鏈

```
問題陳述（Problem Statement）
-> 利害關係人地圖（Stakeholder Map）
-> 需求（Requirements）
-> 使用者故事（User Stories）
-> 驗收條件（Acceptance Criteria）
-> 使用案例 / 資料流程圖 / 實體關係圖（Use Case / DFD / ERD）
-> 軟體需求規格書（SRS）
-> 架構 / 類別圖 / 循序圖 / 介面流程（Architecture / Class Diagram / Sequence Diagram / UI Flow）
-> 軟體設計文件（SDD）
-> 原型（Prototype）
-> 測試案例（Test Cases）
-> 期末展示（Final Demo）
```

如果期末原型（prototype）展示一個功能，學生要能回頭指出：

1. 這個功能來自哪一項需求？
2. 哪個利害關係人（stakeholder）需要它？
3. 驗收條件（acceptance criteria）是什麼？
4. 哪張模型描述它？
5. 測試案例如何驗證它？

### 概念 4：好的專案題目要能被分析、建模、設計與展示

適合本課的題目需要：

1. 有明確使用者與使用情境。
2. 有至少 3 個核心任務流程。
3. 有資料需要管理。
4. 有角色、權限或規則可以分析。
5. 可在 5 週內做出原型展示（prototype demo）。
6. 可用軟體需求規格書（SRS）、軟體設計文件（SDD）、測試案例（test cases）描述清楚。

不適合的題目：

1. 太大：例如「做一個完整校務行政系統」。
2. 太小：例如「做一個倒數計時器」。
3. 太抽象：例如「改善學生生活」但沒有具體流程。
4. 太偏演算法：例如「做一個股票預測 人工智慧系統（AI system）」但沒有清楚使用情境。
5. 無法驗收：例如「做一個比較好用的系統」但沒有判斷標準。

### 概念 5：問題陳述（Problem statement）的四個元素

一個可用的問題陳述（problem statement）至少包含：

| 元素                          | 說明               | 範例                                                                          |
| ----------------------------- | ------------------ | ----------------------------------------------------------------------------- |
| 目標使用者（Target users）    | 誰遇到問題？       | 社團幹部、課外組承辦人、一般學生                                              |
| 目前情境（Current situation） | 現在怎麼做？       | 使用 Google 表單（Google Forms）報名、電子郵件（email）申請場地、人工整理名單 |
| 痛點與影響（Pain / impact）   | 痛點與影響是什麼？ | 場地衝突不易追蹤、候補通知延遲、活動資訊分散                                  |
| 期待改善（Desired change）    | 希望改善什麼？     | 整合活動建立、報名、候補、場地申請與審核流程                                  |

建議句型：

```
對於 [目標使用者（target users）] 而言，目前在 [目前情境（current situation）] 中，常發生 [痛點與影響（pain / impact）]。
本專案希望設計一個 [系統概念（system concept）]，協助 [期待改善（desired change）]，並在本期聚焦於 [範圍焦點（scope focus）]。
```

### 概念 6：範圍（Scope）是設計品質的一部分

範圍（scope）不是把系統做小而已，而是讓團隊知道該把時間用在哪裡。

範圍內（in-scope）範例：

1. 活動建立與編輯。
2. 學生報名與取消報名。
3. 額滿候補。
4. 場地申請與審核。
5. 活動通知狀態顯示。

範圍外（out-of-scope）範例：

1. 真實金流付款。
2. 串接學校正式單一登入。
3. 正式部署到學校伺服器。
4. 手機應用程式（App）原生開發。
5. 完整行政簽核流程。

明確範圍外（out-of-scope）可以避免期末專案失控。

## 範例解析

### 範例 1：從一句模糊需求拆出系統分析問題

原始句子：

```
我們想做一個社團活動管理系統。
```

可以這樣拆解：

| 問題類型 | 可追問問題       | 可能答案                                                             |
| -------- | ---------------- | -------------------------------------------------------------------- |
| 使用者   | 誰會使用？       | 社團幹部、一般學生、課外組、場地管理員、指導老師                     |
| 目標     | 解決什麼問題？   | 報名資訊分散、場地申請流程不透明、通知容易漏掉                       |
| 流程     | 現在怎麼做？     | Google 表單（Google Forms）報名、電子郵件（email）申請場地、人工統計 |
| 資料     | 系統要保存什麼？ | 活動、場地、報名紀錄、候補名單、審核紀錄                             |
| 規則     | 有什麼限制？     | 場地不能衝突、活動需審核、額滿可候補                                 |
| 邊界     | 本期不做什麼？   | 不串接金流、不整合全校單一登入、不做正式部署                         |
| 驗收     | 怎樣算完成？     | 可完成活動建立、報名、候補、場地申請、審核與通知原型（prototype）    |

### 範例 2：把案例問題整理成初步專案題目

較差題目：

```
社團系統
```

可接受題目：

```
社團活動報名與場地申請管理系統
```

更好的題目：

```
面向學生社團幹部與課外組的活動報名、候補與場地申請流程整合系統
```

更好的題目會明確指出：

1. 目標使用者：學生社團幹部、課外組。
2. 核心任務：活動報名、候補、場地申請。
3. 系統範圍：流程整合，不是完整校務系統。

### 範例 3：把模糊題目改成問題陳述（problem statement）

模糊題目：

```
我要做讀書會系統。
```

追問：

1. 誰在管理讀書會？
2. 現在用什麼工具？
3. 最大問題是報名、分組、簽到、資料分享，還是成果追蹤？
4. 哪些事情本期一定要展示？

改寫後：

```
對於大學課程中的讀書會小組與授課老師而言，目前讀書會報名、分組、出席紀錄與閱讀進度常分散在不同表單與訊息群組中，導致老師難以掌握各組進度，學生也不容易確認自己該完成的閱讀任務。
本專案希望設計一個讀書會管理系統，協助老師建立讀書會任務、學生完成分組與簽到，並讓小組回報閱讀進度。本期聚焦於課程內讀書會的分組、簽到與進度追蹤，不處理正式成績系統串接。
```

### 範例 4：專案目標（project goal）與功能（feature）對照

| 不佳寫法                            | 問題                 | 改寫                                         |
| ----------------------------------- | -------------------- | -------------------------------------------- |
| 做登入系統                          | 只是功能             | 讓不同角色依權限進入對應工作流程             |
| 做通知功能                          | 只是功能             | 降低活動變更時通知遺漏的風險                 |
| 做人工智慧推薦（AI recommendation） | 技術先行             | 協助學生根據時間、興趣與活動限制找到適合活動 |
| 做儀表板（Dashboard）               | 使用者介面（UI）先行 | 讓管理者快速掌握待審核、額滿、取消與異常活動 |

### 範例 5：範圍切割

題目：

```
校園二手書交易系統
```

可能範圍內（in-scope）：

1. 賣家刊登書籍。
2. 買家搜尋與收藏。
3. 買賣雙方預約面交。
4. 交易狀態追蹤。
5. 違規或爭議回報。

可能範圍外（out-of-scope）：

1. 線上付款與金流。
2. 物流寄送。
3. 真實身分驗證。
4. 跨校交易。
5. 自動估價演算法。

提醒：

```
範圍外（out-of-scope）不是缺點，而是專案管理能力。
能說清楚不做什麼，代表你知道這 5 週要把什麼做好。
```

## 學生任務

### 任務 1：案例拆解

以小組為單位，針對「社團活動報名與場地申請系統」完成下列分析：

1. 列出至少 5 個利害關係人（stakeholders）。
2. 列出至少 3 個目前流程中的痛點（pain points）。
3. 列出至少 5 種系統需要保存的資料（data）。
4. 列出至少 3 條業務規則（business rules）或限制。
5. 列出至少 2 個可能造成系統失敗的風險（risks）。

輸出格式：

```
小組名稱：

利害關係人（Stakeholders）：
1.社團幹部
2.參與人\學生
3.指導單位
4.場地管理單位
5.系統管理員

痛點（Pain Points）：
1.多頭申請流程複雜，學生須同時向課外活動單位送交計畫書，又須向場地管理單位申請場地，
跨單位流程審核不連貫，審核進度無法透明追蹤。
2.場地資訊不即時透明易發生重複借用或時間衝突，缺乏即時更新的線上查詢系統，
送件時不知道場地是否已被租借。
3.時數核銷與現場簽到 需手動簽名與核對 行政負擔較重
使用外部表單收集報名資訊後，現場簽到多依賴手動勾選紙本名單，
事後彙整參與者名單、發送電子研習證書或計算參與時數時容易出錯且行政負擔重。

資料（Data）：
1.使用者資料
2.社團組織資料
3.場地與設備資源資料
4.活動申請與預約紀錄
5.報名與簽到記錄

業務規則（Business Rules）：
1.所有場地預約與活動辦理申請，必須於活動舉辦日的至少 N 個工作天前送出；活動開始前 N 小時內，申請者無法於系統端自行取消或變更場地，需透過專案申請處理。
2.同一個場地於同一個時間區段（Time Slot）內，絕對互斥（僅能有一筆審核通過的借用紀錄）；且系統需自動驗證申請人身分，唯有「當學期已完成登記備案之社團負責人或指定幹部」帳號方可發起場地預約。
3.若預約場地成功後，無故未進場使用且未依規定辦理取消累計達 2 次（或場地復原不實／破壞設備），系統將自動鎖定該社團之借用權限 30 天或一學期

風險（Risks）：
1.高流量尖峰並發負載風險
2.使用者操作與訓練不足風險
```

完成標準：

- 每一欄都要能對應案例，不可只寫空泛詞。
- 利害關係人（stakeholder）不能只寫「使用者」，要具體。
- 痛點（pain point）需要描述目前問題，不是直接寫解法。

### 任務 2：候選題目發想

每組提出 2-3 個候選題目。每個題目需包含：

1. 系統名稱。
2. 目標使用者。
3. 要解決的問題。
4. 三個核心任務。
5. 系統不處理的範圍。

輸出格式：

```
候選題目 1：
系統名稱：系統家事分工與公費分攤系統
目標使用者：在外合租房客或室友
要解決的問題：合租時可能有水電費或公用物品以及家事之類的事情會無法分工或是錢算不清楚
需要一個可以整合事務之工具
核心任務：
1.成員可新增共同開銷紀錄，系統會自動計算每人應攤提的金額，並彙整出最終結算清單
2.可建立家事輪值表，輪值人員完成任務後可於系統上勾選已完成
3.系統首頁提供個人專屬的看板，顯示「我本週的待辦家事」與「我尚未結清的款項」
本期不處理：串接銀行或 LINE Pay 的實際金流轉帳功能

候選題目 2：
系統名稱:揪團訂飲料幫手
目標使用者：常團訂飲料的人如辦公室或實驗室
要解決的問題：傳統在 LINE 群組喊單或傳閱紙本菜單，
主揪常常需要耗費大量時間統計杯數、冰塊甜度，且極易發生漏訂、算錯錢或團員忘記付款的問題
核心任務：
1.主揪可快速建立一個「專屬訂購房間」，設定收單時間並分享專屬連結給群組成員。
2.成員透過連結進入後，可自行選擇品項、調整冰塊甜度及填寫客製化備註。
3.系統於收單後，自動產出兩種報表：「給店家的總單（彙整各品項總數）」與「主揪的收費清單。
本期不處理：與手搖飲店家的 POS 機直接串接訂單、菜單圖片 OCR 自動辨識建檔、線上信用卡/第三方支付結帳。
```

完成標準：
