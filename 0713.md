# 系統分析與設計：7/13 教材

## 單元名稱

需求工程（requirements engineering）、訪談設計（interview design）、需求來源管理（requirements source management）與程式碼代理實作入門（code agent implementation）

## 課程名稱

系統分析與設計

## 課程定位

1. 問題陳述（problem statement）。
2. 系統範圍（system scope）。
3. 利害關係人分析（stakeholder analysis）。
4. 可行性分析（feasibility analysis）。
5. 專案章程第 2 版（Project Charter v2）。
6. 原型核心任務（core tasks for prototype）。

今天要把「我們想做什麼系統」推進成「需求從哪裡來、如何問出需求、如何記錄需求、如何把需求交給程式碼代理（code agent）做出第一個可操作原型」。

今天完成後，每組應該能交出：

1. 訪談大綱（interview guide）。
2. 模擬訪談紀錄（simulated interview notes）。
3. 需求來源表（requirements source table）。
4. 候選需求清單（candidate requirements list）。
5. 程式碼代理實作任務書（code agent implementation brief）。
6. 第一個可操作切片（first working slice）。
7. GitHub 文件與版本管理平台（GitHub）更新紀錄。
8. 生成式 AI 使用紀錄（generative AI use log）。

## 進度規劃

| 時段  | 主題                                                                                        | 主要產出                                 |
| ----- | ------------------------------------------------------------------------------------------- | ---------------------------------------- |
| 02-04 | 需求工程（requirements engineering）與訪談設計（interview design）                          | 訪談大綱、訪談紀錄、需求來源初稿         |
| 06-08 | 需求萃取（requirements elicitation）、來源管理與程式碼代理實作（code agent implementation） | 需求來源表、實作任務書、第一個可操作切片 |

## 學習目標

完成本單元後，你應能：

1. 說明需求（requirement）與解決方案（solution）的差異。
2. 設計不誘導、不空泛、可追問的訪談問題（interview questions）。
3. 從訪談紀錄或情境資料中萃取需求（requirements elicitation）。
4. 建立需求來源表（requirements source table），讓每項需求都能追溯到來源。
5. 初步區分功能性需求（functional requirement, FR）、非功能性需求（non-functional requirement, NFR）、限制條件（constraint）與業務規則（business rule）。
6. 將需求整理成程式碼代理實作任務書（code agent implementation brief）。
7. 使用程式碼代理（code agent）完成第一個可操作切片（first working slice），但不讓代理替小組決定需求。
8. 在 GitHub 文件與版本管理平台（GitHub）保留需求、提示詞、產出、人工修正與版本紀錄。

## 先備知識

你需要先有：

1. 專案章程第 2 版（Project Charter v2）。
2. 利害關係人分析表（stakeholder analysis table）。
3. 可行性分析摘要（feasibility brief）。
4. 風險清單（risk list）。
5. 至少 3 個原型核心任務（core tasks for prototype）。
6. 小組 GitHub 儲存庫（GitHub repository）。

如果前一週文件尚未完成，今天仍可補做，但需求與程式碼代理實作必須以最新的專案章程第 2 版（Project Charter v2）為準。

## 問題情境

很多小組會從「想做系統」直接跳到「畫畫面」或「請生成式 AI 幫我做一個系統」。這樣通常會出現幾個問題：

1. 系統功能看起來很多，但不知道哪個需求來自哪個使用者。
2. 介面畫面已經做出來，但無法解釋為什麼需要這些欄位。
3. 需求描述像口號，例如「提升效率」「改善體驗」，但無法驗收。
4. 程式碼代理（code agent）做出來的原型很漂亮，但和本組題目的利害關係人、流程、範圍不一致。
5. GitHub 文件與版本管理平台（GitHub）上只有產出，沒有需求來源、提示詞、人工修正紀錄。

今天的重點是：先把需求問清楚，再把需求交給程式碼代理實作。程式碼代理可以加速，但不能取代系統分析與設計的判斷。

## 核心概念

### 概念 1：需求不是功能名稱

需求（requirement）描述的是使用者或系統在特定情境下必須達成的條件，不只是功能名稱。

不夠好的寫法：

```text
活動報名功能。
```

較好的寫法：

```text
一般學生需要在活動名額未滿時完成報名，並在活動名額已滿時看到候補順位與通知狀態。
```

差異在於：

1. 有明確角色。
2. 有情境條件。
3. 有行為。
4. 有結果。
5. 可以在原型或測試中驗收。

### 概念 2：需求要能追溯來源

需求來源（requirements source）可以來自：

1. 訪談（interview）。
2. 問卷（questionnaire）。
3. 觀察（observation）。
4. 現有表單或文件（existing documents）。
5. 同儕角色扮演（peer role-play）。
6. 專案章程（Project Charter）。
7. 利害關係人分析（stakeholder analysis）。

每項需求都應該回答：

1. 這個需求是誰提出的？
2. 是直接說出來，還是從情境推論出來？
3. 它解決哪個痛點？
4. 它是否在本期範圍內？
5. 它能不能驗收？

### 概念 2-1：需求來源表不是紀錄表，而是證據表

需求來源表（requirements source table）的目的不是把訪談內容貼上去而已，而是讓每一個需求都有證據。後面寫使用者故事（user stories）、軟體需求規格書（SRS, Software Requirements Specification）、分析模型（analysis models）、原型（prototype）與測試案例（test cases）時，都要能回頭查到需求來源。

需求來源表至少要做到三件事：

1. 保留原始語意：不要把受訪者的話改到失去意思。
2. 萃取可用需求：從原始摘要中整理出可分析、可設計、可驗收的需求。
3. 標記採用判斷：說明本期是否納入，以及不納入的理由。

需求來源表的每一列都應該像這樣：

| 欄位                              | 用途             | 不建議寫法   | 較佳寫法                                   |
| --------------------------------- | ---------------- | ------------ | ------------------------------------------ |
| 來源編號（source ID）             | 讓需求可追蹤     | 1            | INT-STU-01                                 |
| 來源類型（source type）           | 說明資料來自哪裡 | 學生說的     | 訪談（interview）                          |
| 來源角色（source role）           | 說明誰提供資訊   | 使用者       | 一般學生                                   |
| 原始摘要（source summary）        | 保留原始情境     | 報名很麻煩   | 不知道是否額滿，填完表單後才被通知沒有名額 |
| 萃取需求（extracted requirement） | 寫成可分析需求   | 要有報名功能 | 系統應在活動列表顯示剩餘名額或額滿狀態     |
| 需求類型（requirement type）      | 初步分類         | 功能         | 功能性需求（FR）                           |
| 本期是否納入                      | 控制範圍         | 要           | 是，因為它對應期末原型任務 1               |
| 對應原型任務                      | 連到後續展示     | 無           | 任務 1：學生查看活動並報名                 |

### 概念 2-2：來源編號要有規則

來源編號（source ID）建議用「來源類型 + 角色 + 流水號」命名。這樣後面查資料時比較不會混亂。

| 來源類型                    | 建議前綴 | 範例         | 說明                    |
| --------------------------- | -------- | ------------ | ----------------------- |
| 訪談（interview）           | INT      | INT-STU-01   | 一般學生訪談第 1 筆     |
| 問卷（questionnaire）       | QUE      | QUE-STU-03   | 一般學生問卷第 3 筆     |
| 觀察（observation）         | OBS      | OBS-CLUB-01  | 觀察社團幹部流程第 1 筆 |
| 文件（document）            | DOC      | DOC-FORM-01  | 現有表單或規定第 1 筆   |
| 專案章程（Project Charter） | CHA      | CHA-SCOPE-01 | 專案章程中的範圍限制    |
| 同儕回饋（peer feedback）   | PEER     | PEER-G2-01   | 第 2 組同儕回饋第 1 筆  |

範例：

```text
INT-STU-01：一般學生訪談第 1 筆
INT-CLUB-02：社團幹部訪談第 2 筆
DOC-FORM-01：現有 Google 表單欄位第 1 筆
CHA-SCOPE-01：專案章程範圍限制第 1 筆
```

### 概念 2-3：需求來源也有強弱

不是每一筆來源都一樣可靠。課堂作業可以接受模擬訪談，但仍要知道來源強度不同。

| 來源強度 | 來源例子                                               | 使用方式                             |
| -------- | ------------------------------------------------------ | ------------------------------------ |
| 強       | 真實目標使用者訪談、現有流程文件、正式表單             | 可作為主要需求來源                   |
| 中       | 同儕角色扮演、助教或同學模擬使用者、公開制度說明       | 可作為初步需求，但需標記為模擬或推論 |
| 弱       | 小組自行想像、生成式 AI（generative AI）直接產生的需求 | 只能作為發想，不能直接當作已驗證需求 |

如果需求來自生成式 AI（generative AI）或程式碼代理（code agent），要在來源表中清楚標記，並用人工判斷確認是否合理。

### 概念 3：訪談問題要避免誘導

誘導式問題（leading question）會讓受訪者順著你期待的答案回答，導致需求失真。

| 不建議問題                           | 問題所在           | 較佳問題                                           |
| ------------------------------------ | ------------------ | -------------------------------------------------- |
| 你是不是覺得現在報名很麻煩？         | 已經暗示「很麻煩」 | 你通常怎麼完成活動報名？過程中最花時間的是哪一步？ |
| 如果有自動通知，你會不會覺得很方便？ | 已經暗示解決方案   | 你目前如何知道報名狀態或候補變化？                 |
| 你會想要我們做手機應用程式嗎？       | 直接跳到解決方案   | 你通常在什麼情境下查看活動資訊？用什麼裝置？       |
| 你覺得這個系統是不是應該加入付款？   | 可能擴大範圍       | 報名流程中是否涉及費用？目前費用如何處理？         |

### 概念 4：需求萃取要保留原始語意

需求萃取（requirements elicitation）不是把訪談紀錄改寫成漂亮句子，而是從原始資料中找出需求、限制、例外情況與驗收線索。

範例訪談紀錄：

```text
社團幹部：我們現在用 Google 表單報名，但報名額滿後要手動看誰排候補。有人取消時，我們還要自己去通知下一位，常常會漏掉。
```

可萃取內容：

| 類型                                            | 萃取結果                                     |
| ----------------------------------------------- | -------------------------------------------- |
| 角色                                            | 社團幹部                                     |
| 痛點                                            | 額滿後候補與取消通知需要人工處理，容易漏掉   |
| 功能性需求（functional requirement, FR）        | 系統應在活動額滿後自動將新報名者加入候補名單 |
| 功能性需求（functional requirement, FR）        | 當有人取消報名時，系統應更新候補順位         |
| 非功能性需求（non-functional requirement, NFR） | 候補狀態應清楚顯示，避免使用者誤解           |
| 例外情況（exception case）                      | 報名者取消、候補者放棄、活動名額調整         |

### 概念 5：需求類型要先粗分

今天先做初步分類，後面會再整理成完整需求文件包（Requirements Package）與軟體需求規格書（SRS, Software Requirements Specification）。

| 類型                                            | 中文說明           | 範例                                        |
| ----------------------------------------------- | ------------------ | ------------------------------------------- |
| 功能性需求（functional requirement, FR）        | 系統要做什麼       | 系統應允許社團幹部建立活動並設定名額        |
| 非功能性需求（non-functional requirement, NFR） | 系統要做到什麼品質 | 活動列表應能讓使用者在 3 秒內找到可報名活動 |
| 限制條件（constraint）                          | 本期不可改變的限制 | 本期不串接正式單一登入                      |
| 業務規則（business rule）                       | 流程或組織規則     | 場地申請必須由課外組承辦人核准後才成立      |
| 例外情況（exception case）                      | 正常流程外的狀況   | 活動額滿、報名取消、場地衝突                |

### 概念 6：程式碼代理不是需求負責人

程式碼代理（code agent）可以協助：

1. 依照需求產生原型結構。
2. 建立頁面、表單、假資料與互動流程。
3. 檢查程式碼錯誤。
4. 依照驗收條件（acceptance criteria）修正操作流程。
5. 產生說明檔（README）。

程式碼代理（code agent）不應該替小組決定：

1. 使用者是誰。
2. 哪些需求在範圍內。
3. 哪些需求要刪除。
4. 期末原型要展示什麼。
5. 文件是否可信。

如果程式碼代理產出與需求來源衝突，以需求來源與小組判斷為準。

### 概念 6-1：什麼是程式碼代理

程式碼代理（code agent）是一種可以協助完成軟體開發工作的 AI 工具。它不只回答問題，還可以在得到授權後讀取專案檔案、產生程式碼、修改多個檔案、檢查錯誤、執行指令、整理說明文件，甚至協助建立版本紀錄。

它和一般聊天型生成式 AI（generative AI chatbot）的差異如下：

| 類型                                        | 主要能力                                 | 適合用途                   | 風險                                       |
| ------------------------------------------- | ---------------------------------------- | -------------------------- | ------------------------------------------ |
| 一般生成式 AI 聊天（generative AI chatbot） | 回答問題、整理文字、產生片段             | 發想、改寫、檢查需求矛盾   | 容易產生看似合理但沒有來源的需求           |
| 程式碼補全工具（code completion tool）      | 在編輯器中補程式碼                       | 寫單行或小段程式           | 不一定理解完整需求與範圍                   |
| 程式碼代理（code agent）                    | 讀專案、改檔案、規劃任務、修錯、產生原型 | 從需求任務書做出可操作切片 | 如果任務書不清楚，可能做出錯誤或過大的系統 |
| 低程式碼工具（low-code tool）               | 用圖形介面快速建立表單或流程             | 快速做內部工具或流程展示   | 可能不容易追溯到需求與設計決策             |

在本課中，程式碼代理（code agent）的角色是「協助實作」，不是「替小組做系統分析」。你們仍然要負責需求來源、範圍控制、驗收條件與人工檢查。

### 概念 6-2：目前常見的程式碼代理工具

工具可用性、費用與學校帳號限制可能會變動，使用前請以官方頁面為準。課堂上不要求每組使用同一種工具，重點是能交出需求、任務書、產出與修正紀錄。

| 工具                                                  | 中文定位                                                               | 適合本課的用途                                                     | 注意事項                                                  | 官方參考                                                                      |
| ----------------------------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------ | --------------------------------------------------------- | ----------------------------------------------------------------------------- |
| OpenAI Codex（OpenAI 程式碼代理）                     | 可在瀏覽器、桌面、命令列或整合開發環境中協助讀取、編輯與執行程式碼     | 建立第一個可操作切片、修正錯誤、產生說明檔、檢查需求與程式是否一致 | 要提供清楚任務書與範圍限制                                | https://developers.openai.com/codex                                           |
| Claude Code（Claude 程式碼代理）                      | 可在終端機或開發工具中理解專案、修改檔案、執行測試與處理 Git 工作      | 依照需求修正多個檔案、建立原型、檢查錯誤                           | 使用前要確認授權、費用與資料可否外送                      | https://www.anthropic.com/product/claude-code                                 |
| GitHub Copilot cloud agent（GitHub Copilot 雲端代理） | 可在 GitHub 背景處理任務、研究儲存庫、建立計畫、修改分支與產生拉取請求 | 對已有 GitHub 專案的小組，適合請它處理小功能或文件更新             | 較適合已有 repository 的專案；需理解 branch、pull request | https://docs.github.com/copilot/concepts/agents/cloud-agent/about-cloud-agent |
| IDE 內建代理（IDE agent）                             | 在 Visual Studio Code、Cursor、Windsurf、JetBrains 等開發工具中操作    | 適合邊看程式邊要求修改、解釋或產生檔案                             | 不同工具的功能與限制不同                                  | 依各工具官方文件為準                                                          |

本課建議先選一種工具即可。不要把時間花在比較工具，而是把需求來源表、實作任務書與驗收條件寫清楚。

### 概念 6-3：程式碼代理的輸入品質決定產出品質

程式碼代理（code agent）通常需要以下資訊才容易做出可用產出：

1. 專案背景：這是什麼系統，服務誰。
2. 目標角色：誰會操作，角色任務是什麼。
3. 本次任務：只做哪一個可操作切片。
4. 需求清單：需求編號、需求描述、來源編號。
5. 驗收條件：怎樣算完成。
6. 範圍外項目：哪些東西不能做。
7. 技術限制：是否只能用超文字標記語言（HTML）、樣式表（CSS）、JavaScript 程式語言（JavaScript）、假資料（mock data）。
8. 輸出格式：要它先給計畫，還是直接修改檔案。
9. 檢查方式：要如何確認產出符合需求。

不好的任務：

```text
幫我做一個宿舍修繕系統。
```

比較好的任務：

```text
請根據需求來源表，先做宿舍修繕通報系統的第一個可操作切片。
本次只做：住宿生送出修繕通報，管理者在清單中看到待處理案件。
請使用超文字標記語言（HTML）、樣式表（CSS）與 JavaScript 程式語言（JavaScript），資料用假資料（mock data），不要使用正式登入或資料庫。
完成後請列出每個畫面對應哪些需求編號與驗收條件。
```

### 概念 6-4：實作任務書是給程式碼代理看的小型規格書

程式碼代理實作任務書（code agent implementation brief）不是報告摘要，而是讓代理能執行任務的規格。它要比專案章程更具體，比完整軟體設計文件（SDD, Software Design Document）更小。

一份好的實作任務書至少要回答：

1. 這次只做哪一條流程？
2. 哪些角色會出現在畫面或操作中？
3. 哪些資料欄位一定要出現？
4. 哪些功能不能做？
5. 哪些需求編號一定要被支援？
6. 怎樣操作才算完成？
7. 如果代理想加功能，哪些要拒絕？
8. 產出後要如何檢查？

實作任務書的品質檢查：

| 檢查問題                     | 若答案為否，代表什麼                       |
| ---------------------------- | ------------------------------------------ |
| 是否能指出第一個可操作切片？ | 任務太大，代理可能做出完整但混亂的系統     |
| 是否有需求編號？             | 產出無法追溯需求來源                       |
| 是否有範圍外項目？           | 代理可能加入登入、資料庫、金流等不必要功能 |
| 是否有驗收條件？             | 做完後不知道如何判斷對錯                   |
| 是否有不可捏造事項？         | 代理可能新增不存在的角色、流程或規則       |
| 是否指定技術限制？           | 代理可能使用小組不會操作的技術             |

### 概念 7：第一個可操作切片要小而完整

第一個可操作切片（first working slice）不是整個系統，而是一條可以從開始走到結束的核心流程。

範例：

| 題目                       | 不建議的第一步                 | 建議的第一個可操作切片                   |
| -------------------------- | ------------------------------ | ---------------------------------------- |
| 社團活動報名與場地申請系統 | 先做完整登入、首頁、後台、權限 | 社團幹部建立活動，一般學生看到活動並報名 |
| 校園二手書交易系統         | 先做全部商品分類與聊天功能     | 賣家刊登一本書，買家搜尋並送出預約       |
| 宿舍修繕通報系統           | 先做完整管理後台               | 住宿生送出修繕通報，管理者看到待處理清單 |

一個好的可操作切片需要：

1. 有明確角色。
2. 有起點。
3. 有操作步驟。
4. 有終點。
5. 有可檢查結果。

### 任務 6：撰寫程式碼代理實作任務書

請把本組需求整理成程式碼代理實作任務書（code agent implementation brief）。任務書必須讓程式碼代理知道「要做什麼、不做什麼、怎樣算完成」。

任務書至少包含：

1. 專案背景。
2. 目標角色。
3. 第一個可操作切片（first working slice）。
4. 本期範圍內項目（in-scope）。
5. 本期範圍外項目（out-of-scope）。
6. 需求清單。
7. 驗收條件（acceptance criteria）。
8. 技術限制（technical constraints）。
9. 不可捏造事項。

完成標準：

- 不能只寫「幫我做系統」。
- 至少列出 5 個驗收條件。
- 至少列出 3 個程式碼代理不可自行決定的事項。
- 至少列出 3 個範圍外項目，避免程式碼代理自動擴大功能。
- 至少列出 1 個「若代理想加入，必須拒絕」的功能。

1. 專案背景 (Project Background)
   在校園與辦公室情境中，手搖飲揪團通常依賴通訊軟體（如 LINE 群組）喊單，主揪需耗費大量時間人工抄單、統計冰塊甜度與對帳，極易發生看錯、漏單或代墊虧損。本專案旨在開發一個「線上揪團訂飲料與對帳管理系統」網頁原型，將整個開團、點餐、結單與收款核對流程自動化。
2. 目標角色 (Target Roles)
   主揪 (Host)： 負責發起揪團活動、一鍵結單鎖定、並核對團員付款狀態。

團員 (Participant)： 透過專屬連結進入系統，自主選購飲品並調整客製化備註。

3. 第一個可操作切片 (First Working Slice)
   本專案的第一步是實現 「一個完整且可走通的訂單生命週期」：

主揪成功在網頁建立一個活動 ➡️ 產生專屬網址 ➡️ 團員點擊連結在網頁上點選一杯大杯、微糖、微冰的紅茶並送出 ➡️ 主揪點擊結單鎖定修改 ➡️ 系統自動在同一個畫面上呈現出給店家的「1 杯紅茶（大/微糖/微冰）」總單，以及對帳清單。

4. 本期範圍內項目 (In-Scope)
   主揪建立專屬訂單房間，設定店家名稱與產生邀請連結。

團員獨立選購飲品，並選擇客製化選項（品項、價格、冰塊、甜度、配料）。

團員送出訂單後，前端畫面即時顯示「點餐成功」及訂單明細。

團員在結單前，可於個人點餐頁面修改或刪除已提交的訂單。

主揪一鍵結單功能，按下後鎖定修改權限，團員端無法再新增或編輯。

系統自動計算每位團員的應付金額，並於主揪後台產出收款對帳清單。

系統自動汇总所有品項細節，產出給店家的「品項總數彙總表（店家總單）」。

5. 本期範圍外項目 (Out-of-Scope)
   此處用於規範程式碼代理，避免其自動擴大功能。

與外部真實飲料店家的實體 POS 機進行系統接單串接。

串接真實變動的第三方飲料店家 API（本期僅使用固定的靜態菜單資料）。

線上信用卡、LINE Pay、街口支付等第三方金流支付串接。

實體紙本菜單照片的 OCR（光學字元辨識）自動辨識與上架功能。

手機 iOS 或 Android 原生 App 開發（本期僅需響應式 RWD 網頁）。

6. 需求清單 (Requirements List)
   FR-01 (開團): 主揪可建立房間，並產生唯一 UUID 連結（如 /room/{uuid}）。

FR-02 (選點): 團員進入該連結，可下拉選取飲品及「冰塊（正常/少冰/微冰/去冰）」、「甜度（正常/半糖/微糖/無糖）」與「加料（珍珠/椰果/不加）」並送出。

FR-03 (鎖單): 主揪端按下結單按鈕後，後端將該房間狀態轉為 CLOSED，拒絕任何前端傳入的 POST/PUT 修改請求。

FR-04 (對帳): 系統在結單後，在主揪畫面上依據團員暱稱列表，並自動計算每個人飲料的單價加配料小計金額。

NFR-01 (效能): 頁面載入時間在一般網絡下應小於 2 秒。

NFR-02 (正確): 對帳單的加總金額必須確保 100% 正確，不得有浮點數計算誤差。

7. 驗收條件 (Acceptance Criteria)
   程式碼代理必須滿足以下 5 個條件，方可視為實作完成：

【開團驗收】 主揪點擊建立開團後，資料庫成功寫入一筆新的 OrderSession 紀錄，且前端網址列跳轉為帶有唯一隨機 ID 的房間 URL。

【填單驗收】 團員在房間頁面成功填寫並提交點單後，資料庫寫入一筆 OrderDetail，且前端頁面立刻亮起「點餐成功，您點了：紅茶/微冰/微糖，小計 40 元」之提示，且不需重新整理網頁。

【鎖單防呆驗收】 當主揪按下「結單」後，團員端的「送出/修改」按鈕必須立即變為灰色（Disabled）不可點擊；若團員嘗試使用 Postman 等工具直接向後端發送修改 API 請求，後端必須回傳 403 Forbidden 或 400 Bad Request 的錯誤。

【店家總表驗收】 結單後的店家彙總表，必須正確將相同的產品與客製化進行合併加總（例如：A 點「珍奶/微冰/微糖/50元」、B 點「珍奶/微冰/微糖/50元」，系統報表必須顯示「珍珠奶茶 (微冰/微糖)：2杯，小計 100元」，不得拆開顯示）。

【計算正確性驗收】 模擬 10 筆以上包含加料金額（如珍珠+10元、燕麥奶+15元）的複雜客製化訂單，系統產出的「個人應付金額」與「店家總單大計金額」兩者總和必須 100% 一致。

8. 技術限制 (Technical Constraints)
   資料儲存： 原型階段不需要安裝複雜的 SQL 伺服器，優先使用記憶體內資料庫（In-memory DB）或靜態 JSON 檔案儲存。

菜單限制： 不要設計讓主揪上傳或自行建立菜單的頁面。請預先在後台代碼中 Hard-code（靜態寫死）2 家熱門飲料店（如：50嵐、可不可）的測試菜單。

認證限制： 團員端不需要設計「註冊與登入帳號」功能，團員點餐前僅需輸入「暱稱/學號」作為點餐識別即可。

9. 不可捏造與拒絕事項 (Non-Negotiables)
   以下為程式碼代理「不可自行決定」與「若理想加入必須拒絕」的事項：

A. 代理不可自行決定的事項（必須詢問本組或維持最簡實作）：
資料庫與開發語言框架： 除非本組明確指示，否則禁止自行決定引入複雜的雲端資料庫（如 AWS RDS）。請先使用最基礎的響應式前端（如 HTML/JS/CSS）與輕量後端（如 Python FastAPI 或 Node.js）即可。

菜單品項與客製化細節： 預設的 2 家測試飲料店品項與定價，不可由代理自行發想，必須在與本組確認後，依據我們提供的 JSON 靜態格式進行宣告。

UI 視覺風格與配色： 介面佈局（Layout）與主視覺顏色，代理不可自行隨意美化。請維持最簡潔乾淨的 Wireframe 風格，顏色以黑、白、灰色調與高對比提示色為主。

B. 代理若理想加入，必須拒絕的功能：
【必須拒絕】自動串接 LINE Pay 或街口等行動支付金流： 代理可能會主動提議整合金流 SDK 以增加功能完整度，必須拒絕。本期僅允許主揪在系統上「手動將團員標記為已繳費」，任何線上實際金流轉帳均不在本期原型內。

2. FR-02 (團員可選擇飲品與甜度/冰塊/加料客製化提交)
3. NFR-01 (結單後自動計算總合與店家總單，數據計算無浮點數誤差)

本期範圍內：
- 建立訂餐房間，設定店家並產生唯一分享 URL。
- 團員選購飲品，自訂客製化選項並確認訂單。

- 主揪一鍵結單後，系統將資料庫狀態設為 CLOSED 並阻擋任何修改。
- 系統自動彙總，產出給店家的「品項總單」與給主揪的「收款對帳單」。

本期範圍外：
- 與外部店家的實體 POS 機或菜單 API 進行接單串接。
- 整合 any 線上轉帳金流（如 LINE Pay、信用卡）。
- 設計讓主揪手動新增與修改菜單的後台介面（測試菜單請直接以 JSON 格式靜態寫死於程式碼中）。

- 前端：HTML, CSS (RWD), JavaScript (ES6+)
- 後端：Python (FastAPI)

- 認證：免註冊，團員填單前輸入「暱稱/學號」作為點餐識別即可。

1. 【開團驗收】建立開團後，資料庫成功寫入一筆新的 OrderSession 紀錄，網址跳轉為隨機產生的 UUID 房間網址。

3. 【鎖單防呆】當主揪按下「結單」後，團員端的「送出/修改」按鈕必須立即 Disabled；若直接透過 API 工具發送 POST 請求，後端應拒絕寫入。
4. 【數據計算】模擬 10 筆以上包含加料價格的複雜客製化訂單，系統產出的個人應付金額與店家總計大計金額必須 100% 一致。
